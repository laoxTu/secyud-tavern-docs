# Plugins Module — Usage Guide

## Creating a Client Plugin

### 1. Directory Structure

```
plugins/my-plugin/
├── manifest.json     # Plugin manifest
├── client.tsx        # Client entry point
└── server.ts         # Server entry point (optional)
```

### 2. manifest.json

```json
{
    "id": "my-plugin",
    "version": "1.0.0",
    "clientScript": "client",
    "serverScript": "server"
}
```

- `id`: Unique identifier, also used as the nav tab and API path `[pluginId]`
- `clientScript`: Client entry filename (**without extension**, points to compiled `client.js` from `client.tsx`)
- `serverScript`: Server entry filename (**without extension**, points to `server.ts`)
- `folder` is auto-assigned by `generate-plugin` — do not set manually

### 3. Writing the Plugin

```tsx
// plugins/my-plugin/client.tsx
import { businessNavigationManager } from '@/business/client/navigation';
import React from 'react';

function Content() {
    return (
        <div className="p-8">
            <h1 className="text-xl font-bold">My Plugin</h1>
            <p>Hello from plugin!</p>
        </div>
    );
}

export default function register() {
    businessNavigationManager.register({
        id: "my-plugin",
        label: () => <span>My Plugin</span>,
        component: Content,
    });
}
```

### 4. Build

```bash
# 1. Generate manifests + registerer entry files
npx tsx scripts/generate-plugin.ts

# 2. Build client JS (client plugins only)
npm run build-plugin my-plugin
```

`generate-plugin.ts` performs:
- Scans `plugins/*/manifest.json`
- Generates `src/plugins/manifests.ts` — manifest data
- Generates `src/plugins/server/registerer.ts` — server plugin static imports
- Generates `src/plugins/client/registerer.ts` — client plugin static imports

### 5. Start

```bash
npm run dev
```

After rebuilding, a "My Plugin" tab appears in the navigation bar; click to display the plugin content.

## Creating a Server Plugin

### server.ts

```typescript
// plugins/my-plugin/server.ts
import { pluginRouteManager } from "@/plugins/server/plugin-route";

const route = {
    "my-resource": {
        async GET(request, records) {
            return NextResponse.json({ data: "hello" });
        }
    }
};

export default async function init() {
    pluginRouteManager.registerRouteTree(route);
}
```

Server plugins are imported statically via `@plugins/{folder}/server` (generated by `generate-plugin`). All `@/` references are resolved by Next.js/webpack at build time — no runtime path resolution issues.

### Registering API Routes

Use `pluginRouteManager.registerRouteTree(obj)` to register nested objects as routes:

```typescript
const route = {
    "users": {
        "{id}": {                    // Dynamic parameter {id}
            async GET(request, records) {
                const { id } = records.pluginRouteParams;  // Access dynamic params from records
                return NextResponse.json({ id });
            }
        }
    }
};
pluginRouteManager.registerRouteTree(route);
// Matches GET /plugins/api/users/123 → records.pluginRouteParams = { id: "123" }
```

- HTTP method keys: `GET`, `POST`, `PUT`, `DELETE`
- Dynamic path segments: `{paramName}` format, match any value, extracted to `records.pluginRouteParams`
- All plugin routes go through the `/plugins/api/[...path]` entry point

## Available Imports

Plugins can import host modules via `@/` paths (resolved through the `@plugins/*` alias by webpack):

| Import Path | Exports | Type |
|---|---|---|
| `@/business/client/navigation` | `businessNavigationManager` | TabManager |
| `@/slots/client/conversation` | `conversationManager`, etc. | Registries + utilities |
| `@/components/ui/button` | `Button`, `buttonVariants` | UI components |
| `@/components/ui/dialog` | `Dialog`, `DialogTrigger`, ... | UI components |
| `@/components/ui/*` | All UI components | All available |
| `@/handler/client/error` | `useErrorHandler` | Error handling hook |
| `@/client` | `handleResponse`, `get`, `post`, `put`, `del` | HTTP request utilities |
| `@/plugins/server/plugin-route` | `pluginRouteManager` | Route registration |

Run `npm run gen-stubs` to see which modules are currently registered.

## Internationalization (i18n)

Plugins provide translations in their `localization/` directory, auto-merged into global i18n:

```
plugins/my-plugin/
  localization/
    zh/
      default.json    # { "default": { "my-plugin": "My Plugin" }, "myPlugin": { ... } }
    en/
      default.json
```

Same merge rules as `src/localization/`: the `default` namespace is for generic labels like tab titles; plugin-specific namespaces (e.g. `myPlugin`) are for actual content.

With `useTranslations()`, key paths correspond to the JSON structure: `t('myPlugin.description')` maps to `messages.myPlugin.description`.

## Debugging

- Build artifacts at `plugins/{name}/client.js`
- `generate-plugin` logs scanned plugin lists
- Browser console shows `[plugin manager]` prefixed logs
- Visit `http://localhost:3000/api/plugins/{pluginId}` directly to view the compiled JS
- Visit `http://localhost:3000/api/plugins` to see all registered manifests
- `window.__PLUGIN_API__` available in browser console to inspect all exported modules
